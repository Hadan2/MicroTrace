## MicroTrace 관련

### Q1. 기존 APM(Datadog 등)이 있는데 왜 직접 만들었나요?

> Datadog이나 WhaTap 같은 상용 APM은 비용 최적화를 위해 보통 10초~1분 단위로 메트릭을 집계(Aggregation)하여 저장합니다. 이 때문에 0.1초간 발생하는 마이크로버스트 spike는 평균값에 묻혀 관측이 불가능합니다.
>
> MicroTrace는 이런 상시 모니터링을 '대체'하는 것이 아니라, **트러블슈팅 시 즉각 투입하는 '실시간 프로파일러'**로 기존 APM을 '보완'합니다. eBPF sock_ops로 커널 레벨에서 직접 소켓 단위 latency를 추적하기 때문에, SDK 삽입이나 코드 수정 없이 ms 단위 spike를 잡아낼 수 있습니다.

### Q2. Pixie(New Relic)도 eBPF 기반인데 차이가 뭔가요?

> Pixie는 훌륭한 도구이지만 **Kubernetes 환경에 종속**되어 있어 로컬 Docker 환경에서는 사용할 수 없습니다. 또한 Pixie는 프로토콜 파싱(HTTP, gRPC 등)에 집중하는 반면, MicroTrace는 **spike 감지(sock_ops) → 커널 레벨 원인 추적(kprobe) → 앱 레벨 추적(uprobe)**이라는 3단계 자동 원인 추적 구조를 설계 목표로 두고 있습니다.
>
> 현재 Phase 2(sock_ops 기반 spike 감지)까지 구현이 완료되었으며, "네트워크 문제인가, CPU 스케줄링 문제인가, 디스크 I/O 간섭인가"를 자동으로 좁혀가는 구조는 MicroTrace의 독자적 설계 방향입니다.

### Q3. sock_ops를 선택한 이유는?

> Phase 1에서 kprobe/tcp_connect 방식으로 먼저 구현했습니다. 그런데 testenv에서 HTTP Keep-Alive를 활성화하자 **연결 재사용 구간의 요청이 완전히 추적 불가**한 것을 확인했습니다.
>
> 현대 마이크로서비스는 대부분 Keep-Alive/Connection Pooling을 사용하므로, 이 한계는 치명적이었습니다. sock_ops는 소켓 전체 생명주기를 추적하고, cgroup 단위로 관심 서비스에만 선택적으로 적용할 수 있어 이 문제를 근본적으로 해결합니다.

### Q4. 구현하면서 가장 어려웠던 점은?

> eBPF Verifier와의 싸움이 가장 어려웠습니다. eBPF 프로그램은 커널 안에서 실행되기 때문에 무한 루프, 범위 밖 메모리 접근 등을 Verifier가 정적 분석으로 차단합니다. 복잡한 로직을 넣으면 Verifier가 거부하기 때문에, **"커널에서는 최소한의 데이터 수집만, 복잡한 분석은 유저스페이스(Go)에서"**라는 아키텍처 원칙을 세우게 되었습니다.
>
> 또한 tracepoint의 필드 offset이 커널 버전마다 다른 문제를 format 파일 동적 파싱으로 해결하면서, 리눅스 커널 내부 구조에 대한 깊은 이해를 얻었습니다.

## NetSim Lab 관련

### Q5. tc 명령어를 쓰면 호스트 전체 네트워크가 느려지지 않나요?

> 그게 이 프로젝트의 핵심 기술입니다. Docker 컨테이너는 각각 독립된 **Network Namespace**를 가집니다. Go의 Docker SDK로 컨테이너의 PID를 구하고, `nsenter`를 통해 해당 컨테이너의 네임스페이스 내부로 진입하여 tc 규칙을 적용합니다.
>
> 덕분에 호스트나 다른 컨테이너에는 실질적으로 영향을 주지 않고 타겟 서비스만 격리하여 시뮬레이션할 수 있습니다. 성공 지표로 **"타겟 외 컨테이너 latency 변화 1% 미만"**을 설정하고 검증합니다.

### Q6. Toxiproxy나 Pumba 같은 도구가 있는데 차이점이 뭔가요?

> Toxiproxy는 L7/L4 **프록시 방식**이라 애플리케이션의 엔드포인트를 수정해야 하는 번거로움이 있고, Pumba는 **CLI 중심**이라 실시간 관측이 어렵습니다.
>
> NetSim Lab은 세 가지 차별점이 있습니다:
> 1. **Zero-Code Modification** — 코드 수정 없이 인프라 레벨에서 개입
> 2. **REST API** — CI/CD 파이프라인에서 `curl` 한 줄로 장애 주입/해제 가능
> 3. **MicroTrace 연동** — 장애 주입과 동시에 ms 단위 실시간 관측이 가능한 피드백 루프 제공

### Q7. 왜 iptables가 아니라 tc/netem을 선택했나요?

> iptables는 패킷의 **허용/차단/라우팅**에 최적화된 도구입니다. 반면 latency, jitter, packet loss 같은 **Traffic Shaping**은 tc/netem이 커널의 queueing discipline을 직접 제어하여 구현합니다.
>
> NetSim Lab의 목적은 "패킷을 막는 것"이 아니라 "네트워크 품질을 시뮬레이션하는 것"이므로, tc/netem이 기술적으로 정확한 선택입니다.

## 통합/공통

### Q8. 두 프로젝트를 왜 따로 만들었나요? 하나로 합치면 안 되나요?

> 의도적으로 분리했습니다. **관심사의 분리(Separation of Concerns)** 원칙입니다.
>
> - NetSim Lab은 **장애를 주입**하는 도구 (Active)
> - MicroTrace는 **장애를 관측**하는 도구 (Passive)
>
> 각각 독립적으로도 가치가 있으며, 결합하면 시너지가 생기는 구조입니다. NetSim 단독으로도 로컬 개발자의 디버깅 도구로 쓸 수 있고, MicroTrace 단독으로도 프로덕션 트러블슈팅에 투입할 수 있습니다. 모니터링 도구와 테스트 도구는 독립적으로 발전하되 API로 연동하는 것이 확장성과 유지보수 면에서 유리합니다.

## 압박 질문 대비

### Q9. 스타트업 초기에는 트래픽이 적은데 커널 레벨 프로파일링이 필요한가요?

> 트래픽의 '양'이 아니라 시스템의 '의존성'이 핵심입니다. 사용자가 10명뿐이라도 결제(PortOne)나 AI(OpenAI API) 같은 외부 API에서 3초 타임아웃이 발생하면 서버 스레드가 블로킹되어 연쇄 장애로 이어집니다.
>
> Amazon Builders' Library에서도 재시도(Retry)와 타임아웃 로직의 오작동이 시스템 자멸을 유발한다고 설명합니다. NetSim Lab은 이런 외부 API 지연을 로컬에서 즉시 재현하여 방어 로직을 검증하게 해주고, MicroTrace는 Docker 기반 초기 MSA에서 트래픽 양과 무관하게 발생하는 찰나의 마이크로버스트나 CPU 간섭을 ms 단위로 짚어줍니다.

### Q10. PaaS(Azure App Service, AWS Fargate) 환경에서는 커널 권한이 없어서 못 쓰지 않나요?

> 맞습니다. 권한이 통제된 완전 관리형 PaaS 환경에서는 eBPF 기반의 MicroTrace를 적용할 수 없습니다. 이 점은 명확한 기술적 한계입니다.
>
> MicroTrace의 1차 타겟은 비용 문제로 PaaS를 졸업하고 IaaS(EC2 등)에서 직접 Docker를 운영하는 팀입니다. 상용 APM을 도입할 예산이 부족한 이 구간의 팀에게 무료 오픈소스인 MicroTrace가 최적의 대안이 됩니다. NetSim Lab은 프로덕션이 아닌 **로컬 개발 단계**에서 네트워크 장애를 시뮬레이션하는 용도이므로, 프로덕션 인프라 환경과 무관하게 대부분의 Docker 기반 개발 환경에서 가치를 제공합니다.

### Q11. 로그를 먼저 보면 되지, 왜 인프라/네트워크 레벨까지 내려가나요?

> 애플리케이션 로그는 'API 호출 실패(Timeout)'라는 **결과**는 보여주지만, 그것이 순수 네트워크 지연인지 컨테이너 CPU Throttling인지 **원인**은 알려주지 않습니다. MicroTrace는 네트워크 지표(RTT, 재전송)와 시스템 지표(CPU 포화)를 동일 타임라인에 겹쳐 보여주므로(Correlated Monitoring), 로그가 설명하지 못하는 근본 원인을 찾아냅니다.
>
> OpenTelemetry 같은 분산 추적을 도입하려면 모든 서비스에 SDK를 삽입하고 코드를 수정해야 합니다. 반면 MicroTrace는 sock_ops를 통해 **코드 수정 없이** 커널 레벨에서 즉시 추적을 시작하므로, 전담 SRE가 없는 팀도 즉각적인 트러블슈팅이 가능합니다.

### Q12. Mac/Windows에서 Docker Desktop을 쓰면 eBPF가 제대로 동작하나요?

> 솔직히 말씀드리면, 100% 동작을 보장하기 어렵습니다. Mac(Apple Virtualization)이나 Windows(WSL2)는 경량 Linux VM 위에서 컨테이너를 실행하기 때문에, BTF 지원 여부나 cgroup v2 마운트 방식이 네이티브 Linux와 다를 수 있습니다.
>
> 이 리스크를 관리하기 위해 두 가지 전략을 취하고 있습니다:
> 1. **지원 매트릭스(Support Matrix) 명시** — WSL2 커널 5.15 이상 등 동작 가능 환경을 문서에 명확히 정의
> 2. **검증 환경 분리** — 최종 검증은 EC2 프리티어 등 Native Linux에서 수행하는 것을 최우선 마일스톤으로 설정
>
> NetSim Lab의 경우 컨테이너 내부의 격리된 Network Namespace(veth)에 `nsenter`로 진입하여 tc 규칙을 적용하므로, 호스트 OS의 영향을 상대적으로 덜 받습니다.

## 기술 심화 질문

### Q13. 동적 프로브 활성화 시 관찰자 효과(Observer Effect) 문제는 어떻게 대응하나요?

> 핵심적인 질문입니다. MicroTrace는 평상시 오버헤드를 줄이기 위해 spike 감지 시에만 kprobe/uprobe를 동적으로 활성화하는 구조인데, spike가 발생한 순간은 대상 서버가 이미 극도의 스트레스를 받고 있을 가능성이 높습니다. 이때 커널 훅을 등록하면 추가적인 락 대기와 CPU 인터럽트가 발생하여 오히려 상황을 악화시킬 수 있습니다.
>
> 이를 방지하기 위해 **서킷 브레이커(Circuit Breaker)** 패턴을 설계에 반영하고 있습니다. CPU 사용률이 임계치(예: 95%)를 넘으면 프로브 활성화를 포기하고, sock_ops에서 수집된 기본 메트릭만으로 리포트를 생성하는 Graceful Degradation 전략입니다.

### Q14. Go 런타임 추적(uprobe) 시 ABI 변경에 따른 유지보수 문제는?

> Go는 레지스터 기반 호출 규약 등 내부 ABI를 공격적으로 변경하기 때문에, uprobe 기반 추적에서 바이너리 오프셋이 Go 버전마다 전부 달라지는 문제가 있습니다. Cilium 등 기존 eBPF 도구들도 이 유지보수에 막대한 에너지를 쏟고 있습니다.
>
> 이에 대한 아키텍처 결정으로, 특정 Go 버전을 고정 지원하는 방식 대신 **DWARF 디버깅 심볼을 런타임에 파싱하여 오프셋을 동적으로 계산**하는 방향을 검토하고 있습니다. 다만 이는 Phase 4 계획이며, 현재는 sock_ops 기반의 L4 메트릭 수집의 정확도를 Datadog/Pixie 대비 검증하는 데 집중하고 있습니다.

### Q15. Privileged 권한 요구가 보안 정책과 충돌하지 않나요?

> eBPF를 커널에 적재하려면 CAP_SYS_ADMIN 또는 CAP_BPF, NetSim Lab은 NET_ADMIN이 필요합니다. 결제나 외부 API를 다루는 환경에서 이런 강력한 권한을 부여하는 것에 대한 우려는 당연합니다.
>
> 이를 위해 두 가지 방향으로 설계하고 있습니다:
> 1. **최소 권한 원칙** — 풀 root가 아닌 CAP_BPF + CAP_NET_ADMIN만 부여하는 최소 Capability 설정을 기본 제공
> 2. **감사 추적(Audit Log)** — 도구가 어떤 프로브를 언제 붙이고 떼었는지 로그로 기록하여, 보안 감사 시 투명성을 확보

## 포지셔닝 & 한계 인정

### Q16. 결국 이 도구가 뭔지 한 문장으로 설명해주세요.

> 만능 APM이나 로그 분석기가 아닙니다. PaaS 환경에서는 쓸 수도 없습니다.
>
> **비용 때문에 IaaS(EC2 등)에서 Docker로 마이크로서비스를 운영하는 소규모 팀이, 원인을 알 수 없는 '찰나의 네트워크 지연'과 '타임아웃'을 만났을 때, 코드 수정 없이 커널 레벨에서 가장 정확하게 범인을 찾아주는 특수 목적 프로파일러**입니다.

### Q17. 이 도구가 절대 못 하는 것은?

> 세 가지를 명확히 인정합니다:
>
> 1. **PaaS/Serverless 환경 미지원** — 커널 권한이 없는 완전 관리형 환경에서는 설치 자체가 불가능합니다.
> 2. **비즈니스 로직 로그 분석 불가** — L4/커널 레벨의 TCP 지연과 재전송만 관측합니다. "결제 실패(잔액 부족)" 같은 L7 비즈니스 에러는 ELK 등 로그 시스템의 영역입니다. MicroTrace의 역할은 "이게 네트워크/인프라 문제인지 아닌지"를 1초 만에 O/X로 갈라주는 것입니다.
> 3. **Mac/Windows 네이티브 완벽 지원 불가** — eBPF와 veth 격리는 Linux 커널 기술이므로, 가상화 환경에서는 커널 버전과 설정에 따라 제약이 있을 수 있습니다. 완벽한 동작은 Native Linux에서만 보장합니다.

### Q18. 타겟 페르소나를 구체적으로 정의해주세요.

> **"MSA 전환 초기 단계이거나, 결제/AI 등 외부 API 연동이 필수적인 핵심 비즈니스 로직을 다루는 소규모 백엔드 팀. 상용 APM의 비용이 부담되며, 간헐적인 API 타임아웃의 원인을 찾거나 로컬 환경에서 Retry 로직의 신뢰성을 검증하고 싶은 엔지니어."**
>
> 핵심은 '트래픽의 양'이 아니라 '시스템의 의존성'입니다. 사용자가 10명이라도 외부 결제 API 한 건의 타임아웃이 서비스 전체를 멈출 수 있는 환경이라면, 커널 레벨 프로파일링은 선택이 아니라 필수입니다.
