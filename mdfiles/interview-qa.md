## MicroTrace 관련

### Q1. 기존 APM(Datadog 등)이 있는데 왜 직접 만들었나요?

> Datadog이나 WhaTap 같은 상용 APM은 비용 문제로 보통 10초~1분 단위로 데이터를 평균(Aggregation) 내어 저장합니다. 그래서 0.1초간 발생하는 마이크로버스트 spike는 평균에 묻혀 관측이 불가능합니다.
>
> MicroTrace는 이런 상시 모니터링을 '대체'하는 것이 아니라, **트러블슈팅 시 즉각 투입하는 '실시간 프로파일러'**로 기존 APM을 '보완'합니다. eBPF sock_ops로 커널 레벨에서 직접 소켓 단위 latency를 추적하기 때문에, SDK 삽입이나 코드 수정 없이 ms 단위 spike를 잡아낼 수 있습니다.

### Q2. Pixie(New Relic)도 eBPF 기반인데 차이가 뭔가요?

> Pixie는 훌륭한 도구이지만 **Kubernetes 환경에 종속**되어 있어 로컬 Docker 환경에서는 사용할 수 없습니다. 또한 Pixie는 프로토콜 파싱(HTTP, gRPC 등)에 집중하는 반면, MicroTrace는 **spike 감지 → 커널 레벨 원인 추적(kprobe) → 앱 레벨 추적(uprobe)**이라는 3단계 자동 원인 추적 구조를 가지고 있습니다.
>
> "네트워크 문제인가, CPU 스케줄링 문제인가, 디스크 I/O 간섭인가"를 자동으로 좁혀가는 구조는 MicroTrace만의 독자적 설계입니다.

### Q3. sock_ops를 선택한 이유는?

> Phase 1에서 kprobe/tcp_connect 방식으로 먼저 구현했습니다. 그런데 testenv에서 HTTP Keep-Alive를 활성화하자 **연결 재사용 구간의 요청이 완전히 추적 불가**한 것을 확인했습니다.
>
> 현대 마이크로서비스는 대부분 Keep-Alive/Connection Pooling을 사용하므로, 이 한계는 치명적이었습니다. sock_ops는 소켓 전체 생명주기를 추적하고, cgroup 단위로 관심 서비스에만 선택적으로 적용할 수 있어 이 문제를 근본적으로 해결합니다.

### Q4. 구현하면서 가장 어려웠던 점은?

> eBPF Verifier와의 싸움이 가장 어려웠습니다. eBPF 프로그램은 커널 안에서 실행되기 때문에 무한 루프, 범위 밖 메모리 접근 등을 Verifier가 정적 분석으로 차단합니다. 복잡한 로직을 넣으면 Verifier가 거부하기 때문에, **"커널에서는 최소한의 데이터 수집만, 복잡한 분석은 유저스페이스(Go)에서"**라는 아키텍처 원칙을 세우게 되었습니다.
>
> 또한 tracepoint의 필드 offset이 커널 버전마다 다른 문제를 format 파일 동적 파싱으로 해결하면서, 리눅스 커널 내부 구조에 대한 깊은 이해를 얻었습니다.

## NetSim Lab 관련

### Q5. tc 명령어를 쓰면 호스트 전체 네트워크가 느려지지 않나요? 어떻게 특정 컨테이너만 제어하죠?

> 그게 이 프로젝트의 핵심 기술입니다. Docker 컨테이너는 각각 독립된 **Network Namespace**를 가집니다. 저는 Go의 Docker SDK로 컨테이너의 PID를 구하고, `nsenter`를 통해 해당 컨테이너의 네임스페이스 내부로 진입하여 tc 규칙을 적용합니다.
>
> 덕분에 호스트나 다른 컨테이너에는 전혀 영향을 주지 않고 타겟 서비스만 격리하여 시뮬레이션할 수 있습니다. 성공 지표로 **"타겟 외 컨테이너 latency 변화 0%"**를 설정하고 검증합니다.

### Q6. 이미 Toxiproxy나 Pumba 같은 도구가 있는데 차이점이 뭔가요?

> Toxiproxy는 L7/L4 **프록시 방식**이라 애플리케이션의 엔드포인트를 수정해야 하는 번거로움이 있고, Pumba는 **CLI 중심**이라 실시간 관측이 어렵습니다.
>
> NetSim Lab은 세 가지 차별점이 있습니다:
> 1. **Zero-Code Modification** — 코드 수정 없이 인프라 레벨에서 개입
> 2. **REST API** — CI/CD 파이프라인에서 `curl` 한 줄로 장애 주입/해제 가능 (자동화 테스트 통합)
> 3. **MicroTrace 연동** — 장애 주입과 동시에 ms 단위 실시간 관측이 가능한 피드백 루프 제공

### Q7. 왜 iptables가 아니라 tc/netem을 선택했나요?

> iptables는 패킷의 **허용/차단/라우팅**에 최적화된 도구입니다. 반면 latency, jitter, packet loss 같은 **Traffic Shaping**은 tc/netem이 커널의 queueing discipline을 직접 제어하여 구현합니다.
>
> NetSim Lab의 목적은 "패킷을 막는 것"이 아니라 "네트워크 품질을 시뮬레이션하는 것"이므로, tc/netem이 기술적으로 정확한 선택입니다.

## 통합/공통

### Q8. 두 프로젝트를 왜 따로 만들었나요? 하나로 합치면 안 되나요?

> 의도적으로 분리했습니다. **관심사의 분리(Separation of Concerns)** 원칙입니다.
>
> - NetSim Lab은 **장애를 주입**하는 도구 (Active)
> - MicroTrace는 **장애를 관측**하는 도구 (Passive)
>
> 각각 독립적으로도 가치가 있으며, 결합하면 시너지가 생기는 구조입니다. NetSim 단독으로도 로컬 개발자의 디버깅 도구로 쓸 수 있고, MicroTrace 단독으로도 프로덕션 트러블슈팅에 투입할 수 있습니다.
>
> 이 설계는 실제 소프트웨어 아키텍처에서도 동일한 원칙이 적용됩니다. 모니터링 도구와 테스트 도구는 독립적으로 발전하되, API로 연동하는 것이 확장성과 유지보수 면에서 유리합니다.

1. 동시 접속자가 아닌 '외부 API 의존성'의 문제 (NetSim Lab의 가치)
소규모 팀일수록 모든 것을 직접 개발하지 않고, 외부 SaaS나 API에 절대적으로 의존합니다. 결제(Toss, PortOne), 알림(Slack, Twilio), 최근에는 AI(OpenAI API) 연동이 필수적입니다.

이유 (Why): 사용자가 10명뿐이라도, 결제 모듈이나 외부 LLM API에서 3초의 타임아웃(Timeout)이나 패킷 유실이 발생하면 우리 서버의 스레드(Thread)가 블로킹되고 연쇄 장애(Cascading Failure)로 이어집니다. 소규모 팀은 **"우리 코드가 느린 것"**보다 **"외부 네트워크망이 느려질 때 우리 앱이 어떻게 뻗는지"**를 모르는 경우가 훨씬 많습니다.

근거 및 현실적 타당성: 아마존(AWS)의 유명한 아키텍처 가이드인 Amazon Builders' Library: Timeouts, retries, and backoff with jitter에서는 분산 시스템에서 재시도(Retry)와 타임아웃 로직이 잘못되었을 때 시스템이 어떻게 자멸하는지 설명합니다. NetSim Lab은 바로 이런 외부 API 통신 지연 상황을 로컬에서 tc/netem으로 즉시 재현하여 개발 단계에서 방어 로직을 검증할 수 있게 해줍니다. 동시 접속자 수와 무관하게 시스템의 **'생존성(Resilience)'**을 테스트하는 필수 도구가 됩니다.

2. 마이크로서비스화로 인한 '분산 컴퓨팅의 오류' 노출 (MicroTrace의 가치)
최근에는 소규모 스타트업도 배포와 유지보수의 이점 때문에 모놀리식(Monolithic)이 아닌 초기 형태의 MSA(Docker 기반 다중 컨테이너)나 서버리스를 채택하는 경우가 많습니다.

이유 (Why): 단일 서버에서는 메모리를 통해 즉시 호출되던 함수가, 컨테이너 분리로 인해 네트워크를 타게 됩니다. 이때 개발자들이 흔히 착각하는 것이 "로컬/같은 VPC 네트워크는 항상 빠르고 안정적일 것"이라는 믿음입니다. 이로 인해 찰나의 마이크로 버스트(Micro-burst)나 시스템 자원(CPU, I/O) 부족으로 인한 큐잉 지연을 파악하지 못하고 엄한 애플리케이션 코드만 뒤지게 됩니다.

근거 및 현실적 타당성: 1994년 썬 마이크로시스템즈의 피터 도이치(Peter Deutsch)가 제안한 컴퓨터 과학의 고전 명제인 분산 컴퓨팅의 8가지 오류(Fallacies of distributed computing) 중 첫 번째가 바로 **"네트워크는 신뢰할 수 있다(The network is reliable)"**입니다. MicroTrace는 트래픽이 적더라도, 컨테이너 간 통신에서 발생하는 서브 밀리초(Sub-ms) 단위의 숨은 병목 구간을 식별하여 "네트워크는 신뢰할 수 없다"는 현실을 실시간으로 보여줍니다.

3. 전담 SRE 팀의 부재와 '도입/유지 비용'의 현실
소규모 팀은 인프라와 네트워크를 전문적으로 까볼 수 있는 데브옵스(DevOps)나 SRE(Site Reliability Engineering) 전담 인력이 없습니다.

이유 (Why): 트러블슈팅이 필요할 때, 애플리케이션에 무거운 SDK를 삽입하거나 코드를 수정하는 것은 큰 부담입니다. 또한 Datadog이나 WhaTap 같은 상용 APM은 호스트당 과금 방식으로 소규모 팀에게는 비용 부담이 커서 도입을 망설이게 됩니다.

근거 및 현실적 타당성: eBPF 기술의 최대 장점인 **'Zero-Code Modification(코드 수정 불필요)'**과 **'낮은 오버헤드'**는 전담 인력이 없는 스타트업에게 강력한 무기입니다. 코드 변경 없이 타겟 컨테이너의 PID와 veth 인터페이스만 식별하여 즉각적으로 커널 레벨에서 추적(sock_ops, kprobe)하고 제어(NetSim)할 수 있다는 점은, 상용 툴을 살 돈도, 세팅할 시간도 부족한 스타트업 엔지니어에게 완벽한 대체재가 됩니다.

💡 MT를 위한 전략 제언 (페르소나 뾰족하게 깎기)
지적받은 내용을 방어하기 위해, 타겟 페르소나를 "그냥 작은 웹 스타트업"에서 아래와 같이 구체적으로 디테일 업하시는 것을 추천합니다.

수정된 타겟 페르소나:
"MSA 전환 초기 단계이거나, 결제/AI 등 외부 API 연동이 필수적인 핵심 비즈니스 로직을 다루는 소규모 백엔드 팀. 상용 APM(Datadog 등)의 비용이 부담되며, 간헐적인 API 타임아웃의 원인을 찾거나 로컬 환경에서 재시도(Retry) 로직의 신뢰성을 검증하고 싶은 엔지니어."

이렇게 방어 논리를 세우시면, "사용자가 적으니 필요 없다"는 피드백을 "사용자가 적어도 1개의 요청이 중요한(결제 등) 시스템이나 분산 환경에서는 무조건 필요하다"는 기술적 필연성으로 뒤집으실 수 있을 것입니다. 면접관이나 리뷰어에게 이 논리를 제시하면 시스템의 깊이뿐만 아니라 현업의 실무적 페인 포인트까지 꿰뚫고 있다는 인상을 확실히 줄 수 있습니다!

로컬 환경의 함정: Mac/Windows 환경에서의 OS 종속성 문제
현상: NetSim Lab과 MicroTrace 모두 주요 사용 사례로 "로컬 Docker 환경"을 강조하고 있습니다. 하지만 배포 직후 개발자들로부터 "내 Mac에서는 에러가 나는데요?", "Windows에서는 안 붙습니다"라는 이슈가 쏟아질 확률이 높습니다.

이유 및 근거: 개발자들의 로컬 머신은 순수 Linux가 아니라 macOS나 Windows입니다. 이 환경에서 Docker Desktop은 내부적으로 경량 Linux VM(Apple Virtualization Framework, Hyper-V, WSL2 등)을 띄워 컨테이너를 실행합니다. eBPF sock_ops나 tc/netem은 호스트 운영체제의 커널을 직접 제어해야 하는데, 이 가상화된 커널이 BPF Type Format(BTF)을 지원하지 않거나 cgroup v2 마운트 방식이 다를 수 있습니다.

해결 과제: WSL2 환경에서의 eBPF 미지원 리스크를 인지하고 계신 것처럼, Mac/Windows 로컬 환경에 대한 명확한 지원 범위(Support Matrix)를 정의하거나, 로컬 디버깅용 가상 머신(Vagrant 등)을 함께 배포하는 대안이 필요합니다.

2. 관찰자 효과 (Observer Effect): 동적 프로브 활성화의 위험성
현상: 장애 원인을 파악하려고 MicroTrace가 개입하는 순간, 오히려 대상 서비스가 완전히 다운되는 상황이 발생할 수 있습니다.

이유 및 근거: MicroTrace는 평상시 오버헤드를 줄이기 위해 레이턴시 스파이크 감지 시에만 kprobe와 uprobe를 동적으로 활성화(Attach)합니다. 하지만 레이턴시 스파이크가 발생했다는 것은 대상 서버의 CPU가 100%에 도달했거나 디스크 I/O가 병목에 걸려있는 등 시스템이 극도의 스트레스를 받고 있다는 뜻입니다. 이런 긴급 상황에서 커널 훅(Hook)을 동적으로 등록하고 해제하는 작업은 커널 내부에 추가적인 락(Lock) 대기와 CPU 인터럽트를 유발합니다.

해결 과제: 동적 프로브 부착 실패 시 시스템 안정성을 지키기 위한 'Graceful Fallback' 메커니즘이 절대적으로 필요하며, CPU 사용량이 특정 임계치(예: 95%)를 넘으면 프로브 활성화를 포기하는 '서킷 브레이커(Circuit Breaker)' 로직을 고민해야 합니다.

3. 유지보수 지옥: Go 언어의 지속적인 업데이트와 uprobe ABI
현상: Phase 4로 계획된 Go 런타임 추적 기능 구현 이후, 팀이 새로운 기능 개발은 못 하고 버전 호환성 패치만 전담하게 될 수 있습니다.

이유 및 근거: Go 언어는 컴파일러와 런타임(goroutine 스케줄러 등)을 지속적으로 최적화하며, 레지스터 기반 호출 규약 등 내부 ABI(Application Binary Interface)를 공격적으로 변경합니다. uprobe는 컴파일된 바이너리의 특정 메모리 오프셋을 직접 찔러서 추적하는 방식입니다. 따라서 타겟 마이크로서비스가 Go 1.18로 빌드되었는지, 1.21로 빌드되었는지에 따라 오프셋 테이블이 전부 달라집니다.

해결 과제: Cilium 등 기존 eBPF 도구들의 기술 블로그를 보면 이 오프셋 유지보수에 엄청난 에너지를 쏟고 있습니다. 특정 Go 버전만 고정으로 지원할지, 아니면 DWARF 디버깅 심볼을 런타임에 파싱하여 오프셋을 동적으로 계산할지에 대한 아키텍처 결정이 필요합니다.

4. 보안 정책의 벽: Privileged 권한 요구
현상: 도구의 가치를 인정한 소규모 팀이 이를 프로덕션 환경(EC2 등)에 도입하려 할 때, 인프라 보안 정책에 가로막혀 설치를 포기할 수 있습니다.

이유 및 근거: NetSim Lab은 타겟 컨테이너의 네트워크를 제어하기 위해 최소 NET_ADMIN 기능(Capability)이 필요합니다. MicroTrace 역시 eBPF 프로그램을 커널에 적재하기 위해 CAP_SYS_ADMIN이나 CAP_BPF 등 루트에 준하는 강력한 권한을 요구합니다. 외부 API와 결제 모듈을 다루는 타겟 페르소나 특성상, 권한 탈취 시 시스템 전체가 위험해질 수 있는 Privileged 컨테이너 실행을 꺼릴 수밖에 없습니다.

해결 과제: NET_ADMIN 최소 권한 설계 방향은 매우 바람직합니다. 추가로, 문서 상에 명확한 보안 가이드를 제공하고 "우리 도구는 컨테이너 외부로 데이터를 유출하지 않는다"는 것을 증명할 수 있는 감사 로그(Audit Log)나 코드 오픈소스화 전략이 필요합니다.

공격 1: 트래픽 및 규모 관련
Q. "타겟을 소규모 백엔드 팀으로 잡으셨는데, 스타트업 초기에는 동시 접속자나 트래픽이 적어서 굳이 커널 레벨의 네트워크 프로파일링 도구까지 쓸 일이 없지 않나요?"

방어 핵심: 트래픽의 양(Volume)이 아니라, 시스템의 '의존성(Dependency)'과 '분산 환경'으로 논점을 전환합니다.

A. 방어 답변:

"트래픽이 적더라도 결제(PortOne 등)나 AI 모델(OpenAI API) 같은 외부 API 통신 의존성이 높다면 이야기가 다릅니다. 사용자가 10명뿐이라도, 외부 결제 모듈에서 3초의 타임아웃이 발생하면 서버 스레드가 블로킹되어 연쇄 장애로 이어집니다.
저희의 NetSim Lab은 이러한 외부 API 지연 상황을 로컬 개발 단계에서 docker run 한 줄로 즉각 재현하여, 개발자가 작성한 타임아웃 및 재시도(Retry) 로직이 제대로 작동하는지 검증할 수 있게 해줍니다. 또한, 단일 머신에서 여러 컨테이너를 띄우는 초기 MSA 환경에서는 찰나의 마이크로 버스트나 CPU 간섭이 발생하기 쉬운데, MicroTrace는 트래픽 양과 무관하게 이러한 숨은 병목을 ms 단위로 정확히 짚어줍니다."

🗡️ 공격 2: 클라우드 환경 (PaaS) 관련
Q. "요즘 스타트업은 인프라 관리 인력이 없어서 Azure App Service나 AWS Elastic Beanstalk 같은 PaaS를 많이 씁니다. 이런 완전 관리형 환경에서는 커널 권한이 없어서 eBPF 프로그램을 아예 못 올리지 않나요? 전제 자체가 무너진 것 같은데요."

방어 핵심: PaaS의 한계를 쿨하게 인정하되, 스타트업의 '비용 현실'과 '로컬 개발 환경'을 근거로 타겟의 타당성을 입증합니다.

A. 방어 답변:

"말씀하신 대로 권한이 통제된 PaaS 환경에서는 eBPF 기반의 MicroTrace를 적용할 수 없는 것이 맞습니다. 하지만 저희가 설정한 1차 타겟은 비용 문제로 PaaS를 졸업하고 가장 저렴한 IaaS(EC2 멀티 호스트 환경)로 이주하여 직접 Docker를 운영하는 팀입니다. 상용 APM을 도입할 예산이 부족한 이 구간의 팀들에게 무료 오픈소스인 MicroTrace는 최적의 대안이 됩니다.
더욱 중요한 것은, 프로덕션 환경이 PaaS이든 IaaS이든 개발자들의 로컬 개발 및 테스트 환경은 99% Docker 기반이라는 점입니다. NetSim Lab은 운영 서버가 아니라 이 로컬 개발 단계에 투입되어 복잡한 네트워크 장애를 시뮬레이션하는 용도이므로, 인프라 환경의 제약 없이 모든 백엔드 팀에게 확고한 가치를 제공합니다."

🗡️ 공격 3: 로그 vs 커널 추적 관련
Q. "에러가 발생하면 보통 개발자들은 애플리케이션 로그(비즈니스 로직 로그)를 먼저 뒤집니다. 토폴로지가 얽혀서 문제라면 OpenTelemetry 같은 분산 추적(Distributed Tracing)을 쓰면 되는데, 왜 굳이 인프라/네트워크 레벨까지 내려가야 하나요?"

방어 핵심: 애플리케이션 로그는 '결과'만 알려줄 뿐 '원인'을 알려주지 않으며, 기존 분산 추적 도구의 '도입 비용(코드 수정)'을 찌릅니다.

A. 방어 답변:

"애플리케이션 로그는 'API 호출 실패(Timeout)'라는 결과는 보여주지만, 그것이 순수 네트워크 지연 때문인지, 컨테이너의 CPU 스케줄링 대기(Throttling) 때문인지 원인은 알려주지 않습니다. MicroTrace는 eBPF를 통해 네트워크 지표(RTT, 재전송)와 시스템 지표(CPU 포화)를 동일한 타임라인에 겹쳐서 보여줌으로써(Correlated Monitoring), 로그가 설명하지 못하는 근본 원인을 찾아냅니다.
또한 OpenTelemetry 같은 분산 추적을 도입하려면 모든 서비스의 코드를 수정하고 SDK를 삽입해야 하는 엄청난 비용이 발생합니다. 반면 MicroTrace는 sock_ops를 통해 코드 수정 없이(Zero-Code Modification) 커널 레벨에서 즉시 토폴로지 맵을 그려주므로, 전담 SRE가 없는 팀도 즉각적인 트러블슈팅이 가능합니다."

🗡️ 공격 4: 로컬 OS 종속성 (가장 아픈 기술적 질문)
Q. "로컬 Docker 환경을 강조하셨는데, 개발자들은 대부분 Mac이나 Windows를 사용합니다. Docker Desktop의 경량 Linux VM 환경에서 eBPF나 tc 커널 모듈이 100% 원활하게 동작한다고 보장할 수 있나요?"

방어 핵심: 기술적 리스크를 이미 인지하고 있음을 밝히고(Risk Management), 검증 계획과 대안을 제시합니다.

A. 방어 답변:

"매우 날카로운 지적입니다. 실제로 Phase 1 개발 당시 WSL2 환경에서 eBPF 개발 환경을 구축하며 커널 버전(5.15)과 기능 지원 여부를 꼼꼼히 체크했습니다. Mac(Apple Virtualization)이나 Windows(WSL2)의 가상 커널 환경에서는 BPF 기능이나 cgroup v2 마운트 방식이 제한적일 수 있다는 리스크를 문서에도 명시해 두었습니다.
이를 해결하기 위해, 로컬 환경에서 sock_ops가 미지원될 경우 기존에 구현해둔 kprobe 방식으로 우회(Fallback)하는 전략을 유지하고 있으며, 최종적으로는 EC2 프리티어 등 100% Native Linux 환경에서 검증하는 것을 최우선 마일스톤으로 삼고 있습니다. NetSim Lab의 경우 Linux 기반 컨테이너의 격리된 Network Namespace(veth)에 개입하는 메커니즘이므로, 호스트 OS와 무관하게 컨테이너 내부 네트워크 제어가 가능하도록 설계되었습니다."

1. 완전 관리형 PaaS / Serverless 환경에서는 "절대" 못 씁니다.
팩트: Azure App Service, AWS Fargate, AWS Lambda 같은 환경에서는 인프라(커널) 권한을 클라우드 벤더가 쥐고 있습니다.

한계 인정: "우리 도구는 eBPF와 tc/netem이라는 커널 기술을 쓰기 때문에, 루트(root) 권한이 없는 PaaS 환경에서는 설치 자체가 불가능합니다."

결론: 만약 타겟 고객이 PaaS만 100% 쓰는 팀이라면, 우리 도구는 그들에게 쓸모가 없는 게 맞습니다. 무리해서 "거기서도 쓸 수 있다"고 방어하면 안 됩니다.

❌ 2. 애플리케이션 '비즈니스 로직' 로그는 분석하지 "못" 합니다.
팩트: 지인분이 겪는 고통 중 "결제 실패(잔액 부족)", "비밀번호 불일치" 같은 비즈니스 에러는 L7(애플리케이션 계층)의 데이터입니다.

한계 인정: "MicroTrace는 L4/커널 레벨의 TCP 지연(Latency), 재전송(Retransmission), CPU 포화도만 관측합니다. 페이로드(HTTP 바디 내용)를 까보거나 비즈니스 로직 에러를 추적하는 전통적인 통합 로그 시스템(ELK 스택 등)을 대체할 수 없습니다."

결론: 지인에게 "우리 도구가 네 로그 분석을 다 해줄게"라고 하면 사기입니다. "네가 로그를 뒤지기 전에, 이게 네트워크/인프라 문제인지 아닌지만 1초 만에 O/X로 갈라줄게"가 정확한 포지셔닝입니다.

❌ 3. 로컬 Mac/Windows를 '완벽하게' 네이티브 지원하지는 "않습니다."
팩트: eBPF와 veth 네임스페이스 격리는 뼛속까지 리눅스 커널(Linux Kernel) 기술입니다.

한계 인정: "Mac의 Docker Desktop이나 Windows의 환경에서는 경량 가상화(VM)를 거치기 때문에, 특정 커널 버전이나 cgroup v2 마운트 환경에 따라 100% 동작을 보장하기 어렵습니다. 완벽한 검증과 동작은 Native Linux(Ubuntu 등)나 AWS EC2 같은 환경에서만 보장됩니다."

결론: 개발자 노트북에서 무조건 찰떡같이 돌아간다고 과대 포장하면 안 됩니다. 환경적 제약(WSL2 커널 5.15 이상 등)을 명확히 문서에 박아두어야 합니다.

💡 오히려 한계를 인정할 때 '진짜 무기'가 생깁니다
이렇게 "안 되는 것"을 시원하게 인정해 버리면, 오히려 심사위원이나 현업 개발자들은 고개를 끄덕입니다. **"아, 이 학생이 eBPF의 한계와 클라우드 인프라의 권한 모델을 정확히 이해하고 있구나"**라고 평가가 반전됩니다.

이렇게 선을 긋고 나면 프로젝트의 소개(Elevator Pitch)가 이렇게 바뀝니다:

"저희 도구는 만능 APM이나 만능 로그 분석기가 아닙니다. PaaS 환경에서는 쓸 수도 없습니다.
하지만, 비용 때문에 EC2나 자체 Linux 서버에서 Docker로 마이크로서비스를 굴리는 소규모 팀이, 도대체 원인을 알 수 없는 **'찰나의 네트워크 지연'과 '타임아웃'**을 만났을 때, 단 1ms의 오차 없이 가장 정확하게 범인을 찾아주는 **특수 목적 핀셋(Profiler)**입니다."
