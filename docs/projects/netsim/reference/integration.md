# 🔗 NetSim Lab & MicroTrace 통합/독립 운영 시나리오

> 역할: **레퍼런스(사람용 전문)**. NetSim·MicroTrace 통합 시나리오 상세. AI용 축약은 [`../guide/overview.md`](../guide/overview.md)의 "MicroTrace 통합" 절.

---

## 1. NetSim 단독 운영 모드 (Standalone Mode)

- **목적:** 가볍고 빠른 네트워크 장애 시뮬레이션 및 재현
- **실행:** NetSim 도커 이미지 실행 후 전용 웹 UI(List View) 접속

**프로세스:**
1. **탐색:** 현재 호스트에서 실행 중인 모든 컨테이너를 리스트 형태로 자동 나열
2. **선택:** 장애를 주입할 타겟 컨테이너를 목록에서 선택
3. **제어:** 슬라이더를 통해 Latency, Loss 등을 설정하고 'Inject' 클릭
4. **확인:** UI 내 간이 모니터링 탭을 통해 적용 여부 확인 및 즉시 해제(Rollback)

**활용 단계:** 로컬 개발 단계에서 타임아웃/재시도 로직 검증

## 2. MicroTrace 통합 운영 모드 (Full-Stack Analysis Mode)

- **목적:** 실시간 시각화 기반의 정밀 네트워크 분석 및 카오스 엔지니어링
- **실행:** MicroTrace 대시보드 내에서 NetSim 엔진 활성화

**프로세스:**
1. **시각화:** eBPF 데이터를 기반으로 컨테이너 간 실제 통신 경로를 **토폴로지 맵**으로 표시
2. **연동:** 맵 상의 특정 '노드'나 '연결선(Edge)'을 **클릭하여 NetSim 제어판 호출**
3. **실험:** 장애 주입과 동시에 MicroTrace의 ms 단위 차트로 성능 변화를 실시간 대조
4. **분석:** 주입된 장애가 전체 마이크로서비스 연쇄 반응에 미치는 영향을 한 화면에서 파악

**활용 단계:** 부하 테스트, 운영 장애 재현 및 Root Cause 분석

```
┌─────────────────────────────────────────────────────────┐
│               MicroTrace 대시보드                         │
│                                                           │
│   ┌─────────┐    200ms     ┌─────────┐                   │
│   │ Service │ ──────────→  │ Service │                   │
│   │    A    │    (정상)     │    B    │                   │
│   └─────────┘              └────┬────┘                   │
│                                  │                        │
│                          클릭 → NetSim 제어판 호출         │
│                                  │                        │
│                          ┌───────▼───────┐               │
│                          │  Latency: ━━━━━━ 500ms        │
│                          │  Loss:    ━━━ 5%              │
│                          │  [Inject] [Rollback]          │
│                          └───────────────┘               │
│                                                           │
│   📊 실시간 차트: ▁▂▃▅▇▅▃▂▁ (spike 즉시 관측)            │
└─────────────────────────────────────────────────────────┘
```

## 3. 기술적 공통 메커니즘 (Shared Engine)

두 모드 모두 동일한 핵심 엔진을 공유합니다.

| 메커니즘 | 설명 |
|---|---|
| **Docker API 연동** | 컨테이너 ID를 기반으로 해당 프로세스의 PID 및 veth 인터페이스 식별 |
| **Namespace Traversal** | Go netlink/nsenter를 활용해 타겟 컨테이너의 격리된 네트워크망 진입 |
| **Traffic Shaping** | 커널 netem 모듈을 제어하여 실제 패킷 레벨의 물리적 지연/유실 구현 |

---

## 4. 통합 모드 성공 지표

| 시나리오 | 성공 기준 |
|---|---|
| 토폴로지 맵에서 클릭 → 장애 주입 | **3초 이내** 대시보드에 성능 변화 반영 |
| 장애 주입 → 감지 → 원인 표시 End-to-End | **< 500ms** |
| 연쇄 장애 가시화 | Service B에 장애 주입 시, Service A의 latency 증가가 **맵에서 시각적으로 표현** |

## 5. 사용자 단계별 활용 가이드

| 단계 | 모드 | 활용 |
|---|---|---|
| **개발 단계** | NetSim 단독 | 로컬 컨테이너의 타임아웃/재시도 로직 검증 |
| **부하 테스트 단계** | MicroTrace 통합 | 서비스 간 병목 지점 시각적 확인 및 원인 분석 |
| **운영 분석 단계** | MicroTrace 통합 | 실제 장애 상황을 NetSim으로 재현하고 MicroTrace로 Root Cause 도출 |
