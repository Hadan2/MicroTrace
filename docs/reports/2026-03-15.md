# 2026-03-15 오류 기록

## PID=0, COMM=swapper 문제

### 증상
```
PID: 0    COMM: swapper/7    ->  172.217.213.139:80  latency: 40669 us
```
curl 로 요청을 보냈는데 PID가 0, COMM이 swapper로 찍힘.

### 원인
두 kprobe가 **서로 다른 컨텍스트**에서 실행되기 때문.

```
tcp_connect()            → curl 프로세스 컨텍스트  → bpf_get_current_pid_tgid() = curl PID
tcp_rcv_state_process()  → 네트워크 인터럽트 컨텍스트 → bpf_get_current_pid_tgid() = 0 (swapper)
```

SYN-ACK 패킷이 도착하면 커널이 인터럽트로 처리함.
이 시점엔 curl이 아니라 커널 idle 프로세스(swapper)가 CPU를 점유 중이라
`bpf_get_current_pid_tgid()` 가 의미없는 값(0)을 반환함.

### 해결책
PID와 comm을 `tcp_rcv_state_process` 에서 읽지 말고,
`tcp_connect` 시점에 Hash Map에 타임스탬프와 함께 저장.

Hash Map value 구조체를 추가:
```c
struct connect_info {
    __u64 ts;       // 연결 시작 타임스탬프
    __u32 pid;      // tcp_connect 시점의 PID (curl 등)
    char  comm[16]; // tcp_connect 시점의 프로그램 이름
};
```

`tcp_connect` 에서 저장:
```c
struct connect_info info = {};
info.ts  = bpf_ktime_get_ns();
info.pid = bpf_get_current_pid_tgid() >> 32;
bpf_get_current_comm(&info.comm, sizeof(info.comm));
bpf_map_update_elem(&connect_start, &key, &info, BPF_ANY);
```

`tcp_rcv_state_process` 에서 꺼내서 사용:
```c
struct connect_info *info = bpf_map_lookup_elem(&connect_start, &key);
e->pid        = info->pid;
e->latency_us = (bpf_ktime_get_ns() - info->ts) / 1000;
bpf_get_current_comm 대신 info->comm 사용
```
