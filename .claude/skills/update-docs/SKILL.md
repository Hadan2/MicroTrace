# update-docs — 코드 변경 → 문서 동기화 스킬

## 목적
사용자가 매번 "문서도 갱신해줘"라고 지시하던 것을 **정해진 절차**로 대체한다.
"무엇을 고쳤는가"에서 "무엇을 갱신해야 하는가"를 추론으로 흔들지 않고, 역방향 매핑 표로 **결정론적으로** 잇는다.

## 절차 (순서 고정)

### 1. 변경 집합 수집
- 바뀐 파일을 확인: `git status --porcelain`, `git diff --name-only`.
- 동작·구조·규약에 영향을 주는 파일만 추린다.

### 2. 변경 성격 판정
- **갱신 필요** = 동작이 바뀜 / 새 기능·필드·이벤트 / 데이터 흐름·소유권·식별키 변경 / 외부 인터페이스(포맷·API·프로토콜) 변경.
- **생략 가능** = 오타·리네임·포맷팅·주석·로깅 추가 등 외부에서 관찰되는 동작/규약 불변.
- 애매하면 사용자에게 한 줄로 물어 확정한다(임의 생략·과잉 갱신 금지).

### 3. 대상 문서 결정 (역방향 매핑 — 추론 금지)
- 코드맵(`docs/projects/microtrace/code/microtrace.code.md`)의 **"코드 → 문서 역방향 매핑"** 표에서
  변경 파일/심볼에 해당하는 행을 찾아 갱신 대상 문서를 그대로 가져온다.
- 표에 대응 행이 없는 **신규 기능**이면: `guide/`에 개념 문서 신설 → 코드맵에 새 §추가 →
  역방향 매핑 표에 행 추가 → `.claude/CLAUDE.md` 정방향 표에 행 추가.
- 연관 엣지/데이터 소유권/식별키를 건드렸으면 **`microtrace.edges.md`도** 갱신 대상에 포함한다.
- 진행 단계(Phase·기능 완료)가 바뀌면 `analysis/progress.md`, "지금 하는 작업" 전환이면 `docs/ai/todo.md`.

### 4. 갱신 작성 원칙
- **code 문서**: 파일+심볼 기준으로 최소·정확하게. 줄번호는 적지 않는다(잘 썩음).
- **guide 문서**: 개념/이유 위주. 코드 세부는 code로 미룬다. AI용 축약 자연어 — 길게 풀지 않는다(그건 reference).
- 실제로 바뀐 내용만 반영. 추측으로 보강하지 않는다(검증 안 된 내용은 "확인 필요"로 표기).

### 5. 보고
- 갱신한 문서 목록과 각 핵심 변경을 한 줄씩 요약.
- 갱신을 **생략한** 파일이 있으면 그 이유도 명시 — 누락이 아니라 판단임을 남긴다.

## 하지 말 것
- 역방향 매핑 표를 건너뛰고 "그럴듯한 문서"를 추론으로 고르지 말 것.
- 코드와 무관한 문서를 "정리 차원"으로 같이 손대지 말 것(diff를 작게 유지).
- 줄번호 인용 추가 금지.

## 연계
- **Stop hook**(`check-docs-sync.sh`)이 "코드는 바뀌었는데 docs는 안 바뀜"을 감지하면 이 스킬의 호출 신호가 된다.
  hook은 리마인더만 주입하고, 필요 여부 최종 판단은 이 스킬의 2단계가 한다.
