# 커밋 규약 (microtrace)

> 이 저장소의 커밋 메시지 형식과 절차.
> **MicroTrace 조정**: 템플릿 원본은 "AI는 커밋을 절대 실행하지 않는다"이지만, 이 프로젝트에서는
> **AI가 커밋 메시지를 제시 → 사용자가 승인하면 AI가 `git commit`/`git push`까지 실행**할 수 있다.
> 단 **승인 없이는 절대 커밋·푸시하지 않는다.**

## 핵심 원칙 ★필독★

* **승인 없이 커밋·푸시 금지.** AI는 변경을 정리해 **커밋 메시지를 작성·제시**하고, 사용자가 동의하면 실행한다.
  동의 없이 `git commit`/`git push`를 실행하지 않는다.
* **이슈 없이 작업을 시작하지 않는다.** 새 작업이면 먼저 GitHub Issue 생성을 제안하고 번호를 확인한다.
* **메시지는 코드블록으로 제시**해 사용자가 검토·복사할 수 있게 한다.
* `Co-Authored-By: Claude ...` 같은 **AI co-author 트레일러를 절대 넣지 않는다.** (저장소 Contributor에 AI가 뜨면 안 됨)
* author는 로컬 git 설정이 자동으로 박으므로 **메시지에 작성자 표기를 넣지 않는다.**

## 브랜치 전략 (GitHub Flow + Issue 기반)

1. GitHub Issue 생성(작업 범위·목적 명시) → 번호 확인(예: #12).
2. 브랜치 생성: `feat/#12-cause-kind` / `fix/#15-rtt-bug` / `docs/#20-flow-md`.
   ```bash
   git checkout -b feat/#12-cause-kind
   ```
3. 작업 완료 후 main에 merge(솔로이므로 PR 없이 직접 merge 가능). `closes #번호`로 Issue 자동 close.

## 메시지 형식

```
[분류] 짧은 제목 (closes #번호)      ← 1번째 줄 = 제목. git log --oneline에 이 줄만 보인다
                                     ← 2번째 줄 = 빈 줄 (필수)
파일 수정
- 경로: 무엇을 바꿨는지 한 줄 설명
- 경로: ...

추가
- 경로: 무엇을 추가했는지

제거
- 경로: 무엇을 제거했는지
```

### 규칙
* **제목 줄** = `[분류] 짧은 제목 (closes #번호)`. GitHub 목록·`git log --oneline`에 이 줄만 뜨므로 한눈에 의미가 통하게. 마침표 없이 간결하게.
* **분류**: `[feat]` / `[fix]` / `[refactor]` / `[docs]` / `[chore]` 중 하나. AI가 변경 내용으로 자동 판단해 제안하고, 사용자가 확인·수정한다.
* **`(closes #번호)`**: 해당 Issue가 있으면 제목 끝에 붙인다. 없으면 생략.
* **빈 줄**: 제목과 본문 사이에 반드시 빈 줄 한 줄.
* **파일 섹션**: `git status`/`git diff --staged`로 변경 파일을 **수정(M) / 추가(A) / 제거(D)** 로 분류해 AI가 자동 작성. 각 파일에 **무엇을/왜** 한 줄 설명. 해당 없는 섹션은 생략.

## 예시

```
[feat] spike 원인 자동 판별 구현 (closes #12)

파일 수정
- collector/stats/stats.go: detectCause 추가, spike 시 dst 자원으로 cause 판정

추가
- docs/learning/infra/cause_detection.md: cause_kind 판별 규칙 정리
```

## AI 커밋 절차

작업이 일단락되면(suggest-commit Stop hook이 미커밋 변경을 감지하면) AI는:

0. **자기 작업분만 제안한다.** 이번 세션에서 AI 자신이 수정하지 않은 파일(다른 도구·에디터가 바꾼 것)은
   커밋 메시지를 만들지 않는다. "이 변경은 제가 수정한 것이 아니라 제안하지 않습니다" 한 줄만 남기고 넘어간다.
1. `git status` / `git diff --staged`로 변경 파일을 수집·분류한다.
2. Issue 번호가 있으면 `(closes #번호)`를 제목에 넣는다. (모르면 사용자에게 묻는다)
3. 위 형식대로 **완성된 커밋 메시지를 코드블록으로 제시**한다.
4. **사용자에게 커밋 여부를 묻는다.** 동의하면 AI가 `git commit`(필요 시 `git push`)을 실행한다. 동의 전에는 실행하지 않는다.
5. 사용자가 메시지 수정을 요청하면 수정 후 재확인한다.

> 여러 이슈/관심사가 한 변경에 섞였으면, 의미 단위로 **나눠 커밋**할 것을 제안한다(이슈 단위 = 커밋 단위가 이상적).
