# AI 행동 규칙

> CLAUDE.md가 "필수 참조"로 가리키는 핵심 행동 규칙. 프로젝트 무관 범용 + MicroTrace 특화(사용자 배경).

## 사용자 배경 (★설명 톤 — 반드시 지킴)

- Linux 처음 사용. 기본 명령어(ls, cd, mkdir 등)도 모를 수 있음.
- 명령어를 실행할 땐 **역할과 옵션 의미를 항상 함께 설명**한다.
- 처음 보는 개념은 **비유나 ASCII 그림으로 자세히** 설명한다. 한 줄 요약 금지.
- 개념 설명 순서: **① 이게 뭔지 → ② 왜 필요한지 → ③ 어떻게 동작하는지.**
- 모든 설명은 **한글**로.

## 답변 지침

- **근거 기반** — 코드의 일부만 보고 추론해 답하지 않는다. 더 읽어야 하면 사용자 허락을 받고 읽은 뒤 답한다.
- **확인 안 된 것은 명시** — 클래스/필드/메서드 유무는 반드시 소스를 직접 읽고 답한다. 확인 못 했으면 "확인 필요"라고 적는다.
- **문서를 따라 필요한 코드만** 읽는다. 전체 코드를 통독하지 않는다(작업 순서는 CLAUDE.md).

## 코드 수정 전 연관·소유권 확인 ★필독★

코드를 **수정·추가·버그수정하기 전에** `docs/projects/microtrace/code/microtrace.edges.md`의 해당 행을 먼저 확인한다.

- **외부 소유 데이터(raw)의 원시값을 수정하지 말 것.** 커널(sock_ops)·cgroup·Docker API가 만든 값은 **불변 진실**이다.
  읽기 전용으로 노출되면 외부 소유다. 버그가 나도 raw를 고쳐 증상을 덮지 말고, **우리 코드의 해석/파생값**을 의심한다. (근거: edges.md A표)
- **연관 기능을 함께 검토하라.** 어떤 기능을 건드릴 때 edges.md B표에서 연관 기능을 찾아 함께 본다.
  특히 **빌드가 통과해도 깨지는 의미론적 결합**(`"src→dst"` 키 라우팅, 필드 전파 지도, RTT 단위 해석)은 C표에서 확인한다.
- 새 연관/소유권을 작업 중 발견하면 **edges.md에 한 행 추가**한다.
- **데이터 파일(`*.db`, `*.sqlite`)을 삭제·덮어쓰기 하지 말 것.** `collector/microtrace.db`는 collector가
  latency 히스토리를 7일 보존하는 **사용자 실측 데이터**다. `.gitignore`에 있어도 **빌드 산출물이 아니다.**
  과거 세션이 산출물로 오판해 `rm` 하여 데이터를 소실시킨 사고가 있었다. 검증/벤치 "정리" 중에도 DB는 건드리지 말고,
  꼭 필요하면 사용자에게 확인받는다(백업은 `cp X.db X.db.bak`). `.gitignore` 등록을 "지워도 됨"의 근거로 쓰지 않는다.

## 버그 점검: 추측 금지, 로그 우선 ★필독★

코드만으로 버그의 범인이 **확실히** 잡히지 않으면, 원인을 단정하지 말 것. 임의로 결론 내려 수정하지 말고,
**로그를 먼저 심어 실제 값을 찍은 뒤** 사용자에게 로그를 요청한다. 진단 목적으로 UI/데이터를 검증용으로 함부로 건드리지 말 것.
점검 순서는 edges.md D절을 따른다.

## 검증(Verification) — 빌드 통과 ≠ 작동

코드를 고치고 빌드/유닛테스트가 통과해도, **기능이 실제로 작동하는지는 별개다.** 특히:
- 컴파일러가 안 잡는 **의미론적 충돌**(edges.md C표)은 수정 시 반대편을 사람이 확인한다.
- eBPF attach·컨테이너·EC2 등 **외부 시스템에서만 드러나는 동작**은 사용자가 검증 신호(로그·`make dev` 실행결과)를 줘야 확인 가능하다.
  임의로 "됐다"고 단정하지 않는다.
- **resolver(`ServiceResolver` 구현체)를 건드렸으면 `grep`/코드 존재 확인만으로 "됐다"고 말하지 않는다.**
  `./scripts/verify-resolver.sh`를 실행해 실제 WebSocket 메시지에 IP가 아니라 서비스명이 찍히는지 exit code로 확인한 뒤 보고한다.
  (EC2 없이도 검증되는 이유: 로컬 컨테이너의 실제 IP로 hosts.yaml을 즉석 생성해 EC2와 동일한 코드 경로를 태우기 때문 — 스크립트 상단 주석 참조.)

## 코드 변경 후 문서 동기화

코드의 **동작·구조·규약**을 바꿨으면, 별도 지시가 없어도 `/update-docs` 절차로 영향 문서를 갱신한다.
대상은 코드맵의 "코드 → 문서 역방향 매핑"으로 결정한다. 오타·리네임·포맷팅뿐이면 "문서 갱신 불필요(사유)"를 한 줄 남기고 마친다.
(Stop hook이 코드만 바뀌고 docs가 안 바뀐 경우 리마인드한다.)

## 구현 사다리 — "덜 만들기"

새 코드를 쓰기 전에 위에서부터 멈춘다: ①정말 필요한가(YAGNI)? → ②기존 코드/패턴 재사용 → ③표준 라이브러리 →
④플랫폼 네이티브 → ⑤기존 의존성 → ⑥1줄 → ⑦최소 작성. **삭제 > 추가**, 가장 짧은 작동 diff가 이긴다.
단, MicroTrace는 **교체 예정 경계는 인터페이스로 격리**한다(coding-rules.md 참조) — 이건 예외.
