# C 언어 개념 정리

> eBPF 코드를 이해하는 데 필요한 C 언어 개념들

---

## 1. `__attribute__((preserve_access_index))` 란?

### ① 이게 뭔지
GCC/Clang 컴파일러에게 특별한 지시를 내리는 **컴파일러 힌트(속성)** 입니다.
`preserve_access_index` 는 "이 구조체의 필드 접근 위치(오프셋)를 보존해라" 라는 의미.

### ② 왜 필요한지
CO-RE(Compile Once - Run Everywhere) 동작에 필수입니다.

일반 C 코드에서 구조체 필드에 접근하면:
```c
struct sock *sk;
sk->__sk_common.skc_daddr  // 컴파일 시점에 오프셋이 숫자로 고정됨
                            // 예: "오프셋 32번지"
```

하지만 커널 버전이 다르면 같은 필드가 다른 오프셋에 있을 수 있음:
```
커널 5.15: skc_daddr 오프셋 = 32
커널 6.1:  skc_daddr 오프셋 = 40  (중간에 다른 필드가 추가됐을 수도)
```

`preserve_access_index` 가 붙으면:
- 컴파일러가 "오프셋 32" 대신 "skc_daddr 이라는 이름의 필드" 라는 정보를 .o 파일에 기록
- 런타임에 BTF를 통해 현재 커널에서 skc_daddr 의 실제 오프셋을 동적으로 찾음

```
__attribute__((preserve_access_index)) 없이:
  컴파일 → "오프셋 32에서 읽어라" 고정
  다른 커널에서 실행 → 오프셋 40인데 32 읽음 → 잘못된 데이터

__attribute__((preserve_access_index)) 있이:
  컴파일 → "skc_daddr 필드를 읽어라" 기록
  다른 커널에서 실행 → BTF로 실제 오프셋 확인 → 올바른 데이터
```

### ③ 실제 코드에서
```c
struct sock_common {
    __be32 skc_daddr;
    __be32 skc_rcv_saddr;
    __be16 skc_dport;
    __u16  skc_num;
    __u16  skc_family;
} __attribute__((preserve_access_index)); // ← 구조체 끝에 붙임
```

`BPF_CORE_READ()` 매크로와 함께 사용해야 효과가 있음.
`BPF_CORE_READ()` 없이 직접 `->` 로 접근하면 CO-RE 효과 없음.

## 2. `__uint`, `__be32`, `__u16` 같은 `__` 타입들

### ① 이게 뭔지
리눅스 커널이 정의한 **고정 크기 정수 타입**들입니다.

### ② 왜 필요한지
일반 C의 `int`, `short` 는 플랫폼마다 크기가 다를 수 있습니다:
```
32비트 시스템:  int = 4바이트
64비트 시스템:  int = 4바이트 (이건 같음)
하지만 long:
  32비트 시스템: long = 4바이트
  64비트 시스템: long = 8바이트  ← 다름!
```

네트워크 프로토콜이나 커널 구조체는 **크기가 정확히 고정**되어야 함.
그래서 크기를 이름에 명시한 타입을 사용:

### ③ 타입 목록

| 타입 | 의미 | 크기 |
|------|------|------|
| `__u8` | unsigned 8bit | 1바이트 (0~255) |
| `__u16` | unsigned 16bit | 2바이트 (0~65535) |
| `__u32` | unsigned 32bit | 4바이트 |
| `__u64` | unsigned 64bit | 8바이트 |
| `__s8` | signed 8bit | 1바이트 (-128~127) |
| `__s32` | signed 32bit | 4바이트 |
| `__be16` | big-endian 16bit | 2바이트, 네트워크 바이트 오더 |
| `__be32` | big-endian 32bit | 4바이트, 네트워크 바이트 오더 |

`__be` 접두사는 **big-endian** 을 명시. 포트 번호, IP 주소 등 네트워크 데이터에 사용.

### `__uint` 는 뭔가?

`__uint` 는 BPF Map 선언에서 사용하는 **특수 매크로**입니다:

```c
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);  // type 필드에 값 설정
    __uint(max_entries, 256 * 1024);     // max_entries 필드에 값 설정
} events SEC(".maps");
```

일반 구조체처럼 보이지만 실제로는:
```c
#define __uint(name, val) int (*name)[val]
```
`int (*name)[val]` 형태의 포인터 배열로 변환됨.
이렇게 하면 컴파일러가 BTF 타입 정보에 Map의 속성들을 기록할 수 있음.

## 3. 언더바 2개 `__` 가 앞에 붙는 이유

### ① 이게 뭔지
C 언어의 **네임스페이스 관례(naming convention)** 입니다.

### ② 왜 필요한지
C 언어에는 공식적인 네임스페이스가 없습니다.
모든 변수, 함수, 타입 이름이 같은 공간에 존재하기 때문에 이름 충돌이 날 수 있음.

예:
```c
int size = 10;         // 내 코드의 변수
int size = 20;         // 어떤 라이브러리의 변수 → 충돌!
```

이를 방지하기 위한 관례:
```
__로 시작하는 이름 → 컴파일러/시스템/커널 예약 이름
_로 시작하는 이름 → 표준 라이브러리 예약 이름
일반 이름          → 개발자가 사용
```

### ③ 종류별 의미

```c
__u32              // 커널/libbpf 정의 타입 (시스템 예약)
__attribute__(())  // GCC/Clang 컴파일러 지시어
__be32             // 커널 정의, big-endian 32bit
__BPF_TARGET_MISSING  // 내부 매크로 (직접 사용 X)
```

한마디로: **"이건 내가 만든 게 아니라 시스템/라이브러리가 만든 것"** 이라는 신호.

## 4. `SEC("...")` (섹션) 이란?

### ① 이게 뭔지
실행 파일(ELF) 내부의 **"이름표가 붙은 서랍"** 입니다. 컴파일러에게 "이 코드는 특별한 서랍에 따로 담아줘"라고 지시하는 것입니다.

### ② 왜 필요한지 (CCTV 비유)
커널에 이미 있는 `tcp_connect` 같은 함수를 수정하는 것이 아닙니다. 
1. **관찰 카메라(eBPF 코드)**를 만든 뒤,
2. **"이 카메라는 tcp_connect 문 앞에 설치해줘"**라고 적힌 **주소표(SEC)**를 붙여서,
3. **로더(libbpf)**에게 전달하는 것입니다.

로더는 섹션 이름을 보고 "아, 이 코드는 커널의 이 지점에 연결(Hook)해야겠구나!"라고 판단합니다.

### ③ 섹션 이름 종류
```c
SEC("kprobe/tcp_connect")       // tcp_connect 함수 입구에 CCTV 설치
SEC("tracepoint/tcp/tcp_retransmit_skb")  // TCP 재전송 이벤트에 CCTV 설치
SEC(".maps")                    // BPF Map 설정 정보 서랍
SEC("license")                  // 라이선스 정보 서랍
```

### ④ SEC 매크로 정의
```c
#define SEC(name) __attribute__((section(name), used))
// section(name): 이 코드를 "name" 섹션에 배치
// used: 컴파일러 최적화로 이 코드가 제거되지 않도록 보호
```

### ⑤ 실제 코드에서

```c
// Map 선언 - .maps 섹션에 배치
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 256 * 1024);
} events SEC(".maps");           // ← .maps 섹션

// kprobe 프로그램 - "kprobe/tcp_connect" 섹션에 배치
SEC("kprobe/tcp_connect")
int BPF_KPROBE(handle_tcp_connect, struct sock *sk) { ... }

// 라이선스 - "license" 섹션에 배치
char LICENSE[] SEC("license") = "GPL";
```

libbpf 가 .o 파일을 읽을 때:
- `.maps` 섹션 발견 → "BPF Map 이구나" → Map 생성
- `kprobe/tcp_connect` 섹션 발견 → "kprobe 프로그램이구나" → tcp_connect에 훅 연결
- `license` 섹션 발견 → "GPL 라이선스구나" → GPL 전용 커널 함수 사용 허가

### ⑥ 섹션 종류에 따른 로더(libbpf)의 동작 차이

로더(`libbpf`)는 섹션의 이름뿐만 아니라 그 안에 담긴 것이 **코드**인지 **데이터**인지에 따라 완전히 다르게 동작합니다.

| 구분 | 함수 섹션 (`SEC("kprobe/...")` 등) | 맵 섹션 (`SEC(".maps")`) |
|:---:|:---|:---|
| **대상** | 실행 가능한 **코드 (함수)** | 설정 정보가 담긴 **데이터 (구조체)** |
| **로더의 해석** | "이 코드를 커널의 특정 지점에 **연결(Attach)**해라" | "이 설정대로 커널에 **자원(Map)을 생성**해라" |
| **비유** | 관찰 카메라를 설치할 **위치 주소표** | 요리를 주문할 **메뉴 주문서** |
| **결과** | 특정 이벤트 발생 시 함수가 실행됨 | 데이터를 저장/공유할 공간이 생김 |

## Union (공용체) 이해하기

### ① 이게 뭔가요? (정의)
`union`은 여러 개의 변수가 **동일한 메모리 주소(공간)**를 공유하도록 만드는 C 언어의 문법입니다. 

일반적인 `struct`(구조체)는 멤버들이 기차 칸처럼 나란히 서 있다면, `union`은 멤버들이 **하나의 의자**에 번갈아 앉는 것과 같습니다. 즉, 의자는 하나인데 그 의자를 "A의 의자"라고 부르기도 하고 "B의 의자"라고 부르기도 하는 셈입니다.

**구조체(struct) vs 공용체(union) 비유:**
```text
[ struct (기차 칸) ]          [ union (한 지붕 다가구) ]
┌──────────┬──────────┐      ┌────────────────────┐
│  거실(A) │ 침실(B)  │      │   거실 겸 침실(A/B) │
└──────────┴──────────┘      └────────────────────┘
 메모리: A크기 + B크기          메모리: A와 B중 큰 쪽
```

### ② 이게 왜 필요한가요? (커널에서의 용도)
리눅스 커널에서 `union`을 사용하는 이유는 크게 세 가지입니다.

1.  **메모리 절약:** 수백만 개의 소켓 객체를 관리하는 커널에서 단 몇 바이트라도 아끼는 것은 전체 시스템 성능에 큰 영향을 줍니다.
2.  **다양한 관점 제공 (주된 이유):** 같은 데이터를 때로는 "32비트 숫자 2개"로, 때로는 "64비트 숫자 1개"로 보고 싶을 때 사용합니다.
3.  **하위 호환성:** IPv4와 IPv6처럼 서로 크기가 다른 데이터를 같은 필드 자리에서 처리하기 위해 사용합니다.

### ③ 어떻게 동작하나요? (비유와 예시)

**비유: 가변형 주차 칸**
> "하나의 커다란 주차 칸이 있습니다. 여기에는 경차(IPv4) 4대를 촘촘히 세울 수도 있고, 커다란 트럭(IPv6) 1대를 세울 수도 있습니다. 주차 관리자(커널)는 상황에 따라 이 칸을 '경차 칸'으로 부르기도 하고 '트럭 칸'으로 부르기도 합니다."

**`tcp_trace.bpf.c`에서의 예시:**
`struct sock_common` 내부의 `union`은 목적지 IP와 출발지 IP를 겹쳐놓았습니다.

```c
struct sock_common {
    union {
        struct {
            __be32 skc_daddr;      // 목적지 IP (4바이트)
            __be32 skc_rcv_saddr;  // 출발지 IP (4바이트)
        };
        __u64 skc_addrpair;        // IP 두 개를 합친 8바이트 데이터
    };
    // ...
};
```

*   **`skc_daddr`을 읽을 때:** 메모리의 앞부분 4바이트만 쏙 빼서 봅니다.
*   **`skc_addrpair`를 읽을 때:** 앞뒤 4바이트를 통째로 합쳐서 커다란 8바이트 숫자로 한꺼번에 봅니다.

---

### Union과 eBPF CO-RE의 만남
커널 버전마다 `struct sock`의 모양이 조금씩 다를 수 있습니다. CO-RE를 통해 eBPF 프로그램이 커널에서 무사히 실행되려면:
1.  **닮은꼴 선언:** eBPF 코드에 적어주는 구조체는 커널 원본과 **생김새(이름, 필드명, union 구조 등)**가 최대한 닮아야 합니다.
2. 우리가 코드에 `union`을 굳이 적어주는 이유는, 커널에게 **"내가 찾으려는 데이터의 구조가 원본 커널과 이렇게 닮았어!"**라고 신호를 보내어 정확한 오프셋(위치)을 계산하도록 돕기 위함입니다.
