# 03.29 트러블슈팅

---

## 1. skel.h가 구버전이라 `handle_sock_ops` 필드 없음

### 증상
IDE에서 `skel->progs.handle_sock_ops` 에 빨간 줄:
```
구조체 "<명명되지 않음>"에 "handle_sock_ops" 필드가 없습니다.
```

### 원인
`tcp_trace.skel.h`는 `bpftool gen skeleton`으로 자동 생성되는 파일.
`.bpf.c`를 수정해도 `make`를 다시 돌리기 전까지는 이전 버전 그대로 유지됨.
skel.h 안에는 여전히 Phase 1의 `handle_tcp_connect` 등이 등록되어 있었음.

### 해결
```bash
make clean && make
```
`.bpf.c` 수정 후엔 항상 빌드를 먼저 돌려야 함.

## 2. sock_ops에서 `bpf_get_current_pid_tgid()` 사용 불가

### 증상
```
libbpf: prog 'handle_sock_ops': BPF program load failed: Invalid argument
unknown func bpf_get_current_pid_tgid#14
커널 로드 실패: -22
```

### 원인
eBPF는 프로그램 타입마다 사용 가능한 헬퍼 함수가 커널에서 화이트리스트로 관리됨.
`bpf_get_current_pid_tgid()`, `bpf_get_current_comm()` 은 sock_ops 타입에서 사용 불가.

```
kprobe 타입  → bpf_get_current_pid_tgid() ✅ 허용
sock_ops 타입 → bpf_get_current_pid_tgid() ❌ 금지
```

sock_ops는 소켓 이벤트 시점에 호출되는데, 이 시점에 "현재 프로세스" 컨텍스트가
유효하지 않을 수 있어서 커널이 아예 막아놓음.

### 해결
`pid` 필드에 `skops->local_port` 를 대신 저장.
`comm` 필드는 `__builtin_memset`으로 빈 문자열 처리.

```c
e->pid  = skops->local_port;  // 소켓 식별용
__builtin_memset(e->comm, 0, sizeof(e->comm));
```

## 3. `BPF_SOCK_OPS_RTT_CB` 가 발생하지 않음

### 증상
`BPF_SOCK_OPS_RTT_CB` 케이스를 추가했는데 Keep-Alive 요청마다 `[RTT]` 이벤트가 안 나옴.

### 원인
`RTT_CB`는 기본적으로 비활성화 상태.
소켓 연결 수립 시점(`ACTIVE_ESTABLISHED_CB`)에 커널에 명시적으로 플래그를 세팅해야
이후 요청마다 RTT_CB가 발생하기 시작함.

### 해결
`ACTIVE_ESTABLISHED_CB` 케이스 안에 플래그 세팅 추가:

```c
case BPF_SOCK_OPS_ACTIVE_ESTABLISHED_CB: {
    // 이 소켓에서 RTT_CB 활성화
    bpf_sock_ops_cb_flags_set(skops,
        skops->bpf_sock_ops_cb_flags | BPF_SOCK_OPS_RTT_CB_FLAG);
    ...
}
```

## 4. `remote_port` 변환 버그 (포트가 0 또는 36895로 출력)

### 증상
```
[RTT]  lport: 47674  ->  127.0.0.1:0      ← 포트 0
[RTT]  lport: 52858  ->  127.0.0.1:36895  ← 포트 36895 (8080이어야 함)
```

### 원인
`remote_port`의 실제 저장 방식을 잘못 파악해서 변환 방식을 세 번 바꿈.

디버그용으로 `remote_port` 원시값을 `latency_us`에 찍어서 확인:
```
rtt: 2417950720  →  0x901F0000
```

`0x901F0000` 을 분해하면:
```
0x901F = htons(8080) = 8080의 network byte order 표현
0x901F0000 = 0x901F << 16
```

즉 `remote_port`는 `(network_byte_order_port << 16)` 형태로 저장됨.

### 시도한 방법들
```c
bpf_ntohl(skops->remote_port) >> 16  // ❌ 결과: 36895
skops->remote_port >> 16             // ❌ 결과: 36895 (bpf_ntohs 빠짐)
bpf_ntohs(skops->remote_port)        // ❌ 결과: 0 (>> 16 빠짐)
```

### 해결
```c
// >> 16 으로 포트를 하위 16비트로 내린 뒤, bpf_ntohs()로 바이트 오더 변환
e->dport = bpf_ntohs(skops->remote_port >> 16);  // ✅ 결과: 8080
```

### 교훈
커널 헤더 주석(`/* Stored in network byte order */`)이 실제 저장 방식과 다를 수 있음.
변환 버그는 원시값을 직접 출력해서 16진수로 분석하는 게 가장 빠름.

```c
// 디버그 패턴: 의심스러운 값을 latency_us 같은 u64 필드에 임시로 찍기
e->latency_us = skops->remote_port;  // 원시값 확인용
```
