# 03.31 트러블슈팅 — Docker SDK import 경로 혼란

## 배경

collector를 패키지 단위로 분리하면서 `resolver/resolver.go`에서 Docker API를 사용했다.
Docker Go SDK가 최근에 모듈 구조가 크게 바뀌어서 에러가 3단계에 걸쳐 발생했다.

---

## 에러 1 — `types.ContainerListOptions` undefined

```
# microtrace/collector/resolver
resolver/resolver.go:98:55: undefined: types.ContainerListOptions
resolver/resolver.go:129:45: undefined: types.EventsOptions
```

### 원인

인터넷 예제 코드는 대부분 **구버전 Docker SDK** 기준이다.

구버전(v20 이전):
```go
import "github.com/docker/docker/api/types"

cli.ContainerList(ctx, types.ContainerListOptions{})
cli.Events(ctx, types.EventsOptions{})
```

신버전(v28+):
```go
// ContainerListOptions, EventsOptions가 types 패키지에서 사라짐
// → client 패키지 자체로 이동
import dockerclient "github.com/moby/moby/client"

cli.ContainerList(ctx, dockerclient.ContainerListOptions{})
cli.Events(ctx, dockerclient.EventsListOptions{})  // 이름도 바뀜
```

### 핵심 변화 요약

| 항목 | 구버전 | 신버전 |
|---|---|---|
| 모듈 이름 | `github.com/docker/docker` | `github.com/moby/moby/client` (분리됨) |
| 옵션 타입 위치 | `api/types` 패키지 | `client` 패키지 안으로 이동 |
| Events 옵션 이름 | `types.EventsOptions` | `client.EventsListOptions` |
| Events 반환값 | `(chan Message, chan error)` | `EventsResult{Messages, Err}` (단일 구조체) |

## 에러 2 — ambiguous import (모듈 중복)

```
go: github.com/moby/moby/client: ambiguous import: found package in multiple modules:
    github.com/moby/moby v28.5.2+incompatible
    github.com/moby/moby/client v0.3.0
```

### 원인

Docker/Moby 프로젝트가 **모노레포에서 멀티모듈**로 전환 중이다.

```
과거 구조 (모노레포):
  github.com/docker/docker  ← 하나의 거대한 모듈
      /api/types/...
      /client/...

현재 구조 (멀티모듈):
  github.com/moby/moby/client  v0.3.0  ← client만 별도 모듈
  github.com/moby/moby/api     v1.54.0 ← API 타입만 별도 모듈
  github.com/moby/moby         v28.5.2+incompatible ← 구버전 호환용 (전체 포함)
```

`go get github.com/moby/moby`(전체)와 `go get github.com/moby/moby/client`(분리된 모듈)를
동시에 go.mod에 넣으면 Go가 "같은 패키지 경로인데 어디서 가져올지 모르겠다"고 충돌한다.

### 해결

incompatible 버전을 제거하고 분리된 모듈만 사용:

```bash
go mod edit -droprequire github.com/moby/moby
go mod edit -droprequire github.com/docker/docker
go get github.com/moby/moby/client@v0.3.0
go get github.com/moby/moby/api@v1.54.0
```

## 에러 3 — `net.IPAddress` 타입 불일치

```
resolver/resolver.go:108:25: invalid operation: net.IPAddress != "" 
    (mismatched types netip.Addr and untyped string)
```

### 원인

구버전 Docker SDK에서 IP 주소는 단순 `string`이었다.

```go
// 구버전
type EndpointSettings struct {
    IPAddress string  // 그냥 문자열
}
```

신버전에서는 Go 표준 라이브러리의 `netip.Addr` 타입으로 바뀌었다.

```go
// 신버전 (moby/moby/api v1.54.0)
type EndpointSettings struct {
    IPAddress netip.Addr  // net/netip 패키지의 값 타입
}
```

`netip.Addr`는 문자열이 아니기 때문에 `!= ""` 비교가 컴파일 에러다.

### netip.Addr란?

Go 1.18에서 추가된 IP 주소 전용 값 타입(`net/netip` 패키지).
기존 `net.IP`(바이트 슬라이스)보다 메모리 효율적이고 비교 연산이 안전하다.

```go
// 유효성 확인
if addr.IsValid() { ... }   // "" 비교 대신

// 문자열 변환
ip := addr.String()          // "172.17.0.3" 형태
```

### 해결

```go
// 변경 전 (구버전 스타일)
if net.IPAddress != "" {
    r.cache[net.IPAddress] = name
}

// 변경 후 (신버전)
if net.IPAddress.IsValid() {
    ip := net.IPAddress.String()
    r.cache[ip] = name
}
```

## 정리 — 신버전 Docker SDK 사용 시 체크리스트

1. `go get github.com/moby/moby/client@v0.3.0` 사용 (docker/docker 아님)
2. 옵션 타입은 `client.ContainerListOptions{}` 등 client 패키지 안에 있음
3. Events는 `client.EventsListOptions` + 반환값 `EventsResult.Messages`, `.Err`
4. IP 주소 필드는 `netip.Addr` → `.IsValid()`, `.String()` 사용
5. `github.com/moby/moby` (incompatible 전체) 와 `github.com/moby/moby/client` 동시 사용 금지
